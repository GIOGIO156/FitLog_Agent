# 方法说明

## 目的

本文档解释 FitLog Local 为什么采用当前这套饮食、碳水策略和运动消耗方法。它面向想先理解 App 方法逻辑、再决定是否信任这些数字的用户。

FitLog 是记录与估算工具，不提供医疗建议，不诊断疾病，不开饮食处方，也不能替代专业人士。

文中的 `[REF-ALG-01](References.md)` 这类标记，对应 [References](References.md) 中的条目；那里记录完整来源和证据边界。

## 核心思路

FitLog 把三件经常被混在一起的事拆开：

1. 目标阶段：用户是在减脂还是增肌。
2. 基础计算方法：每日目标是 `kcal` 优先，还是宏量克数优先。
3. 策略层：在基础目标算出后，是否再重分配碳水，或者做滚动复盘。

这样拆分很重要，因为不同用户真正拿来执行的是不同的主目标。有的人先看 `kcal`，有的人先看蛋白质、碳水、脂肪的克数。如果把两套系统混成一个公式，用户会更难判断到底哪个数字才是主信号。

## `energy_ratio`：kcal 优先的计划方式

`energy_ratio` 适合希望把每日 `kcal` 目标、已摄入和剩余量当作主信号的用户。

它的流程是：

```text
BMR 估算
-> 由共享训练频率得到默认非运动基线
-> 减脂赤字或增肌盈余
-> 加上已记录的净运动消耗
-> 按百分比分配到蛋白质/碳水/脂肪
```

为什么需要这个方法：

- 很多饮食计划从能量平衡开始：减脂吃低于维持，增肌吃高于维持 [REF-ALG-07](References.md)。
- 当主目标是 `kcal` 时，宏量百分比比较容易理解；按热量百分比分配宏量也是常见的营养规划框架 [REF-ALG-04](References.md)。
- 因为基础线刻意设成“不包含专门训练的日常基线”，所以已记录运动可以作为额外可摄入量加回去。

用户需要知道：

- `diet_goal_phase = cutting` 时，`daily_energy_goal_kcal` 会被当成赤字。
- `diet_goal_phase = bulking` 时，`daily_energy_goal_kcal` 会被当成盈余。
- 在这个模式里，`kcal` 目标、摄入和剩余是主计数器。
- 宏量克数是由 `kcal` 目标和宏量百分比换算出来的。
- BMR 和默认生活系数只是估算，不是精确测量 [REF-ALG-01](References.md), [REF-ALG-02](References.md)。
- 共享训练频率设置是两种饮食模式共同使用的用户输入，但在 `energy_ratio` 里，只要本地校准积累了足够历史，运行时仍会优先使用校准出的系数。

主要证据背景：

- BMR/RMR 估算：[REF-ALG-01](References.md), [REF-ALG-02](References.md)
- 宏量热量换算：[REF-ALG-03](References.md)
- 按热量百分比分配宏量：[REF-ALG-04](References.md)

## `gram_per_kg`：宏量优先的计划方式

`gram_per_kg` 适合希望把蛋白质、碳水和脂肪克数当作主目标的用户。

它的流程是：

```text
体重
-> 目标阶段
-> 性别选项
-> 粗略训练频率档位
-> 蛋白质/碳水/脂肪 g/kg 表
-> 宏量克数目标
```

为什么需要这个方法：

- 很多训练导向用户会用每公斤体重多少克来理解蛋白质、碳水和脂肪 [REF-ALG-05](References.md), [REF-ALG-06](References.md)。
- 蛋白质和碳水需求常常比单一 `kcal` 数字更自然地跟体型和训练背景相关 [REF-ALG-06](References.md), [REF-ALG-15](References.md)。
- 当用户更关心“今天把宏量吃到位没有”时，宏量优先更容易执行。

用户需要知道：

- `gram_per_kg` 不使用 BMR、活动水平、每日赤字/盈余、已记录运动热量或宏量百分比。
- `training_frequency_per_week` 只是粗略查表档位：每周 2、3、4 或 5 天。
- 这个档位不是对训练强度、训练年龄、总训练量或表现需求的精确判断。
- `prefer_not_to_say` 会使用同档位男女平均值。
- 在这个模式下，宏量克数是主目标。
- `kcal` 只是辅助信息，因为 `protein*4 + carbs*4 + fat*9` 只是宏量目标的能量等价，不是驱动计划的主计数器 [REF-ALG-03](References.md)。

为什么这里 `kcal` 是辅助：

如果 FitLog 先用 `g/kg` 算出宏量，再强行让这些宏量服从另一套独立的 `kcal` 目标，两套系统就可能互相打架。用户会看到宏量目标在说一件事，`kcal` 计数器又在说另一件事。FitLog 通过让宏量克数成为 `gram_per_kg` 的唯一主目标，避免这种混乱。

主要证据背景：

- 宏量热量换算：[REF-ALG-03](References.md)
- 蛋白质 `g/kg` 范围和运动营养背景：[REF-ALG-05](References.md), [REF-ALG-06](References.md)
- 饮食与身体成分框架：[REF-ALG-07](References.md)

## 为什么两种饮食模式不能混算

这两种模式回答的是不同问题。

`energy_ratio` 回答：

```text
给定我的 kcal 目标，我应该吃多少克蛋白质/碳水/脂肪？
```

`gram_per_kg` 回答：

```text
给定我的体重和训练背景，我应该瞄准多少克蛋白质/碳水/脂肪？
```

两者都可能有用，但不应该同时控制同一个主目标。FitLog 把它们分开，是为了让用户始终知道哪个数字才是真正的“主目标”。

## 碳循环

`carb_cycling` 是减脂期的策略层。它会在基础目标已经算出之后，重新分配一周里的碳水。

它的流程是：

```text
基础碳水
-> 标记高 / 中 / 低碳日
-> 对 7 天倍率做归一化
-> 某些天提高碳水，某些天降低碳水
-> 控制整周平均碳水
```

为什么需要这个方法：

- 有些人希望把更多碳水放在训练更重的日子，把更少碳水放在轻松日或休息日。
- 碳水需求会随训练需求变化 [REF-ALG-15](References.md)。
- 每周归一化可以避免把“碳循环”变成悄悄多吃，或者悄悄压得太低。

什么时候设高碳、中碳、低碳：

- 高碳日通常更适合一周里最重、最长、或者最看重表现的训练日，例如最重的下肢训练日、训练量最大的复合动作日，或者需要明显训练输出的日子 [REF-ALG-15](References.md), [REF-ALG-21](References.md)。
- 中碳日通常适合普通训练日：仍然需要一定供能，但不是全周最吃力的那天。
- 低碳日通常更适合休息日、恢复日、技术性训练日，或者明显更轻松的活动日。
- 这是一条实用分配规则，不是说所有人都必须用同一个模板。FitLog 的目标是让用户能把碳水供应和训练需求大致对上，而不是再引入一套更复杂的新算法。

用户需要知道：

- 碳循环不是神奇减脂算法；周期化碳水限制的证据不应该被过度夸大 [REF-ALG-14](References.md)。
- 如果整周总摄入和执行稳定性不好，它不会自动带来更好的结果。
- 它保持蛋白质和脂肪相对稳定，主要调整的是碳水。
- 它有安全下限：碳水不应低于 `max(weightKg * 1.2, 100)`。
- 如果触发下限，FitLog 会直接限住目标，并记录本地原因码。
- 当前倍率是 FitLog 的本地设置层。它会先把基础碳水往上或往下拨动，再对整周做归一化，避免平均值偏得太离谱。
- 一个常见起点是：
  最重的训练日设高碳，普通训练日设中碳，休息日或轻松日设低碳。
- 如果恢复、饥饿感控制或执行稳定性变差，优先调整日类型，而不是继续把倍率拉得更激进。

主要证据背景：

- 周期化碳水可用性：[REF-ALG-13](References.md)
- 周期化碳水限制证据边界：[REF-ALG-14](References.md)
- 碳水需求随训练需求变化：[REF-ALG-15](References.md)
- 以 `g/kg` 表达碳水、并按训练需求灵活分配的实务背景：[REF-ALG-21](References.md)

## 碳水渐降

`carb_tapering` 是减脂期里的本地复盘策略。它不会替用户自动节食。

它的流程是：

```text
复盘近期体重趋势
-> 检查饮食记录覆盖率
-> 检查训练稳定性
-> 比较当前减重速度和目标区间
-> 建议保持、降低碳水、暂停渐降或不动作
-> 等待用户确认
```

为什么需要这个方法：

- 减脂过程里经常需要小幅调整，但静态的体重变化规则本身就有限制 [REF-ALG-11](References.md), [REF-ALG-19](References.md)。
- 体重会受到水分、食物体积、钠、消化和训练压力影响。
- 滚动复盘比盯着某一天称重更稳，因为它能减少短期噪音的影响 [REF-ALG-19](References.md), [REF-ALG-20](References.md)。
- 用户确认可以避免 App 在后台悄悄把计划越收越紧。

减重速度和渐降步长怎么设：

- 比较保守、也最适合多数人的默认起点，通常是每周体重下降约 `0.5%`。这和常见的体型运动减脂保肌建议比较一致 [REF-ALG-17](References.md)。
- 往 `1.0%/周` 靠拢会更激进，只有在饮食记录很完整、恢复状态稳定、训练也没有明显下滑时才更适合 [REF-ALG-17](References.md)。
- 比 `0.5%/周` 更慢也不是错，尤其是已经比较瘦、训练恢复很敏感，或者单纯想把减脂做得更稳的人。FitLog 允许这样做，但把它视作用户选择，而不是“所有人都该这样”。
- 渐降步长本身是 FitLog 的本地保守规则，不是文献直接规定的固定处方。App 之所以偏向小步调整，是因为单日体重噪音不小，而且趋势还没站稳时不应该频繁猛砍 [REF-ALG-19](References.md), [REF-ALG-20](References.md)。
- 实操上：
  如果你当前碳水已经不高、记录质量一般，或者已经接近碳水下限，更适合从 `5-10 g/天` 这种小步长开始。
- 实操上：
  只有当趋势明显偏慢，而且记录和训练数据都比较扎实时，才更适合考虑 `15-20 g/天` 这种更大的步长。

复盘周期怎么选：

- 当你的体重日波动很大时，用 `14-28 天` 的窗口会更稳 [REF-ALG-20](References.md)。
- `7 天` 会反应更快，但前提是称重和饮食记录都非常稳定，否则更容易误判。
- 所以 FitLog 把复盘周期看成“稳定性控制旋钮”，而不是宣称某一个天数对所有人都最科学。

用户需要知道：

- FitLog 看的是滚动趋势，不是单次称重。
- 饮食记录覆盖率很重要，因为记录不足会让趋势难以解释。
- 训练稳定性很重要，因为训练明显下降时，继续激进降碳可能不合适。
- 建议的碳水变化是小步调整，不是自动惩罚。
- 如果数据不足，App 应该给出 `no_data`，而不是假装知道。
- 如果减重太快，App 可以建议 `pause_taper`，这符合 App 对减重速度的保守 framing [REF-ALG-17](References.md)。
- 如果碳水会低于安全下限，App 会阻止继续下降。
- 渐降偏移是累积的。当前碳水目标大致可以理解为：
  `base carbs + current taper delta`
- 所以一次 `10 g` 的建议，不只是“今天少 10 g”，而是把当前站立中的碳水目标再往下挪一步，直到用户接受后续复盘，或者主动改回设置。

主要证据背景：

- 训练阶段的蛋白质保留：[REF-ALG-16](References.md)
- 保守减重速度框架：[REF-ALG-17](References.md)
- 备赛期宏量变化观察：[REF-ALG-18](References.md)
- 动态体重变化模型的限制：[REF-ALG-19](References.md)
- 日常体重的短期波动背景：[REF-ALG-20](References.md)

## 为什么运动消耗使用净消耗

FitLog 会把已记录运动消耗加到 `energy_ratio` 的可摄入目标里。为了避免重复计算，这里的运动消耗必须是“额外净运动消耗”。

原因很简单：

每日基础线已经包含身体在那段时间即使休息也会消耗的热量。如果 FitLog 直接加上总运动消耗，而不扣除静息消耗，就会把同一份静息热量算两次。

有氧计算使用：

```text
netMet = max(0, MET - 1)
netCardioKcal = netMet * 3.5 * bodyWeightKg / 200 * durationMinutes
```

对于内置和自定义有氧，FitLog 可以保存本次记录实际使用的强度依据。记录页不会只问“轻松/中等/困难”这种模糊主观标签，而是询问用户保持同样速度或节奏大概能连续维持多久。选项会映射到本地 MET：

- 60 分钟以上：低强度。
- 30-60 分钟：中等强度。
- 10-30 分钟：较高强度。
- 3-10 分钟：高强度。
- 小于 3 分钟，需要休息：极高强度/间歇。

最后一个选项会更谨慎处理。间歇训练通常包含休息时间，所以 FitLog 会要求填写实际运动分钟数，并用实际运动分钟数参与 MET 计算，而不是把整段经过时间都当成极高强度。

为什么减去 `1 MET`：

- `1 MET` 大致代表静息能量消耗 [REF-ALG-09](References.md)。
- 非运动基线已经包含静息代谢。
- 减去 `1 MET`，就是为了估算“做这个活动比休息多消耗了多少”。

例子：

如果某个活动是 `8 MET`，FitLog 会把额外运动成本近似看成 `7 MET`，而不是 `8 MET`，因为最初那 `1 MET` 已经被算进每日基线。

主要证据背景：

- MET 数值与换算：[REF-ALG-08](References.md), [REF-ALG-09](References.md)

## 为什么力量训练不按分钟线性算

力量训练不适合简单建模成“每分钟多少 `kcal`”。

原因：

- 60 分钟力量训练可能包含重组、轻组、热身和很长的组间休息。
- 两次时长相同的训练，负荷、次数和总训练量可能完全不同；训练需求本来就会因情境而大幅变化 [REF-ALG-06](References.md)。
- 如果按分钟线性计算，会高估休息时间，也会低估真正的训练工作量。

FitLog 使用训练量和动作类型：

```text
有效负荷
-> 次数
-> 总训练量
-> 动作 profile 系数
-> 主动举铁消耗
-> 恢复和适应成分
```

力量动作可以有不同的录入口径。有些哑铃或双边绳索动作按每侧重量填写，有些单臂或单腿动作按每侧次数填写，自重动作可能填写额外加重，辅助动作填写辅助重量。FitLog 会保留用户原始输入用于展示，同时保存标准化后的计算值用于训练量和热量估算，避免把一只哑铃误当成总重量，或把单侧次数误当成整组总次数。

自定义力量动作沿用同一套思路。用户选择的是容易理解的录入口径，例如总重量、每侧重量、自重加重、辅助重量、总次数、每侧次数或单组时长；App 再把这些选择映射到现有本地力量 profile，而不是让用户直接选择内部 profile 名称。

对于辅助引体向上或辅助双杠臂屈伸，记录的重量是辅助重量，所以 FitLog 会先按 `体重 - 辅助重量` 估算实际移动负荷，再套用同一套力量训练启发式。

时长仍然有作用，但只作为有上限的恢复密度修正。它可以让更密集和更慢的 session 出现小幅差异，但不会线性增加热量。

用户需要知道：

- 力量消耗估算是 FitLog 的本地启发式。
- 它适合在 FitLog 内保持一致，不是实验室级测量。
- App 优先避免明显重复计算，而不是假装能精确知道运动消耗。

## 为什么 FitLog 要有边界和确认

FitLog 使用本地确定性规则和用户确认，是因为输入数据本身并不完美。

重要的不确定来源：

- 餐食估算可能有误差。
- 外部 AI 估算可能有误差。
- 体重变化可能来自水分、食物体积、钠、训练压力或消化状态；静态体重变化模型本身就有限制 [REF-ALG-11](References.md), [REF-ALG-19](References.md)。
- 运动消耗估算是近似值 [REF-ALG-08](References.md), [REF-ALG-09](References.md)。
- BMR 和活动系数是人群估算 [REF-ALG-01](References.md), [REF-ALG-02](References.md)。

FitLog 对不确定性的处理方式：

- 保持计算模式分离
- 使用滚动窗口，而不是单日反应
- 限住不安全或过激的数值
- `carb taper` 必须由用户确认
- 展示策略原因和上下文，而不是把它藏起来
- 除非真的实现了 AI，否则不宣称 App 内有 AI
